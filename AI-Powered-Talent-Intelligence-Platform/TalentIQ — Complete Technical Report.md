## 1. Project Overview

### Project Name
**TalentIQ** — AI-Powered Talent Intelligence Platform

### Project Purpose
TalentIQ is a full-stack, SaaS-grade Applicant Tracking System (ATS) that uses Google's Gemini 2.5 Flash AI to automatically score, rank, and analyze job candidates against job requirements. It gives recruiters an intelligent, data-driven hiring workflow rather than manual resume sifting.

### Business Problem Solved
Traditional ATS tools either reject candidates with rigid keyword filters (causing talent loss) or overwhelm recruiters with hundreds of unscored resumes. TalentIQ solves this by:
- Automatically parsing resumes and scoring every applicant on skills, experience, education, and keyword match using AI
- Providing visual kanban pipeline management so stages are visible at a glance
- Offering a self-serve candidate portal so candidates can track their application status
- Generating actionable AI weekly insights about pipeline health and cross-job candidate matches

### Target Users
1. **Recruiters** — Create jobs, review AI-scored candidates, manage pipeline
2. **Hiring Managers** — Schedule/conduct video interviews, submit scorecards
3. **Admins** — Manage team, settings, company pipeline, billing plan
4. **Viewers** — Read-only access to pipelines and analytics
5. **Candidates** — Apply via public career portal, track status, receive offers and hire letters

### Main Value Proposition
> *"Stop losing great candidates to outdated filters. TalentIQ scores every applicant with explainable AI."*

Every resume uploaded is automatically parsed and scored (0–100) against the job requirements. The recruiter sees not just a number, but a breakdown of skills match, experience match, education match, keyword match, individual strengths, and gaps — all AI-generated.

### Industry Category
**HR Technology (HRTech) / SaaS / Applicant Tracking System**

### Real World Use Cases
1. A startup posts a Senior React Engineer job. A candidate applies. TalentIQ parses their resume via Gemini, extracts skills (React, TypeScript, Node.js), and scores them 84/100 against the job. The recruiter sees the candidate at the top of the Kanban board.
2. A recruiter schedules a video interview. The candidate receives portal access. The interview happens in TalentIQ's built-in WebRTC video room. The interviewer submits a scorecard afterward.
3. A candidate is selected. The recruiter generates an AI-written hire letter, sends it. The candidate signs it via the portal. A notification fires to the recruiter.
4. Every Monday, the AI generates a weekly talent summary: pipeline health score, total hired, avg AI score, and cross-job matches (candidates who are stalled in one job but highly qualified for another).

## 2. Executive Summary

TalentIQ is a production-ready, multi-tenant SaaS ATS built with Next.js 14 App Router on a MongoDB + Mongoose backend, fully integrated with Google Gemini 2.5 Flash AI. The platform combines features typically found in enterprise ATS tools (Greenhouse, Lever, Ashby) with modern AI capabilities in a single Next.js monorepo.

### What the Application Does
- Companies register, create jobs, and publish them to a public career portal
- Candidates apply through a branded portal, upload resumes which are parsed and scored by Gemini AI in real time
- Recruiters manage candidates through a drag-and-drop Kanban pipeline
- Interviews are scheduled, conducted via in-app WebRTC video call, and scored via structured scorecards
- Offers and formal hire letters (AI-generated) are sent to candidates through the portal
- Recruiters receive real-time notifications and AI-generated weekly analytics summaries

### Core Functionality
- AI resume parsing + multi-dimensional scoring (4 categories)
- Kanban pipeline with drag-and-drop (dnd-kit)
- Interview scheduling with conflict detection (working hours, no double-booking)
- WebRTC video calling with DB-backed signaling (polling-based, no external service needed)
- Candidate portal with token-based authentication
- Offer management with PDF export via html2pdf.js
- AI hire letter generation
- AI weekly analytics summary with cross-job matching engine
- Real-time notifications (polling via React Query)
- Full-text search across jobs, candidates, applications
- Rich text editor for job descriptions (Tiptap)

### Key Innovations
1. **DB-backed WebRTC signaling** — Instead of a persistent Socket.io server for signaling, signals are posted to MongoDB (with 24-hour TTL) and the frontend polls every 2 seconds. This eliminates the need for a long-running dedicated signaling server in production.
2. **Cross-job AI matching** — The analytics engine automatically identifies candidates who applied for one role but are highly qualified (≥75% skill match) for another open role in the same company.
3. **Explainable AI scoring** — Gemini returns not just a number but structured JSON with strengths, gaps, and per-reason positive/negative flags that the UI renders visually.
4. **Multi-tenant by design** — Every model has `companyId`, every API route reads the JWT's `companyId` claim. No cross-company data leakage.
5. **Dual auth system** — Separate JWT systems for recruiter/company users (httpOnly cookies + localStorage) and candidate portal users (candidate_token cookie).

### Technical Complexity
This is a genuinely complex full-stack application. The system orchestrates: 2 AI services (Gemini for scoring/parsing/writing), 2 file upload services (UploadThing, ImageKit), a real-time notification layer, WebRTC peer connections, multiple background tasks (Inngest), dual authentication systems, a multi-tenant data model across 17 Mongoose schemas, and a public-facing career portal — all from a single Next.js 14 codebase.

### Overall Architecture
**Unified Next.js Monorepo** with:
- Next.js App Router for all pages and API routes
- Separate Node.js Express + Socket.io process for WebRTC signaling fallback
- MongoDB Atlas as the primary database
- Zustand for global client state
- TanStack Query for server state / caching
- Google Gemini 2.5 Flash for all AI tasks
- Inngest for background job queueing (email sending)
- UploadThing for resume/file uploads
- ImageKit for avatar/image CDN

### Feature: User Authentication (Recruiter)
**Description:** Email + password login with bcrypt password hashing and JWT access/refresh token pair. Google OAuth via google-auth-library.  
**Business Purpose:** Secure recruiter and admin access to the ATS.  
**Technical Implementation:**
- `POST /api/auth/login` — verifies bcrypt password, issues 60-min access token + 7-day refresh token via httpOnly cookies
- `POST /api/auth/google` — verifies Google ID token via OAuth2Client, auto-creates company + user on first login
- `POST /api/auth/register` — creates Company + User in a single transaction
- `POST /api/auth/refresh` — validates refresh token, issues new access token
- `POST /api/auth/logout` — removes refresh token from DB array, clears cookies
- `GET /api/auth/me` — returns current user + company info from JWT claim
- Middleware in `middleware.ts` protects all app routes at edge level
**Files Responsible:** `src/app/api/auth/*/route.ts`, `src/core/auth/jwt.ts`, `middleware.ts`  
**Dependencies:** `bcryptjs`, `jsonwebtoken`, `google-auth-library`, `jose`  
**Database Models:** User, Company

### Feature: Cross-Job AI Matching
**Description:** The analytics engine finds candidates who applied for one role but have a ≥75% skill match with a different open job at the same company. These "cross-matches" are surfaced to recruiters as actionable insights.

### Feature: WebRTC Video Calling
**Description:** In-app video interview room with local/remote video streams, microphone toggle, camera toggle, screen sharing. Uses database-backed signaling (no persistent socket needed).  
**Business Purpose:** Eliminates need for external video tools (Zoom, Google Meet). Interview happens inside the ATS.

- A separate `socket-server/index.js` Express + Socket.io server exists for real-time signaling (WebSocket alternative, started concurrently with `npm run dev`)
- `meet/[interviewId]/page.tsx` is the video room page (currently uses local media only — no peer connection in this file; peer connection logic lives in `useWebRTC.ts`)
**Files Responsible:** `src/hooks/useWebRTC.ts`, `src/app/meet/[interviewId]/page.tsx`, `src/app/api/interviews/room/[roomId]/signal/route.ts`, `socket-server/index.js`

### State Management Architecture

TalentIQ uses a **dual state management** approach:

1. **Zustand** (client-side global state, in-memory):
   - `auth.store.ts` — current user, loading state, fetchUser action
   - `domain.store.ts` — interviews, messages, offers, users, settings (also holds demo mode data)
   - `candidates.store.ts` — candidate list with filters
   - `jobs.store.ts` — job list
   - `notifications.store.ts` — notification list + unread count
   - `ui.store.ts` — sidebar state, mobile nav

2. **TanStack Query** (server state, caching):
   - Hooks: `useJobs`, `useJob`, `useApplications`, `useApplication`, `useInterviews`, `useAnalytics`, `useNotifications`, `useBilling`
   - Query client configured in `lib/QueryProvider.tsx`

### Authentication Pattern
Every protected API route uses the same pattern:
```typescript
let token = req.headers.get('authorization')?.split(' ')[1];
if (!token) token = req.headers.get('cookie')?.split(';').find(c => c.trim().startsWith('accessToken='))?.split('=')[1];
if (!token) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 });
let decoded;
try { decoded = verifyAccessToken(token) as any; } catch (e) { return NextResponse.json({ error: 'Invalid token' }, { status: 401 }); }
```

The `decoded.companyId` claim is then used to scope every database query, ensuring multi-tenant isolation.

### Database Connection
`src/core/database/mongoose.ts` implements a singleton connection with global caching via `global._mongooseClientPromise` to prevent multiple connections in serverless (Next.js) environments.

### Error Handling
Consistent pattern across all routes: try/catch with `console.error` logging and generic `{ error: 'Internal server error' }` response with 500 status. No centralized error handling middleware. Error specificity varies by route.

## 10. Database Analysis

### Collections Overview

#### User
**Purpose:** Recruiter/admin accounts linked to a company.  
**Key Fields:** name, email, passwordHash, googleId, refreshTokens[], role (`admin|recruiter|hiring-manager|viewer`), company (ref: Company), plan (`starter|growth|enterprise`), isActive, resetPasswordToken.  
**Relationships:** Belongs to Company. Referenced by Interview (interviewers), Application (assignedTo), Scorecard (interviewer).

#### Company
**Purpose:** Multi-tenant entity. Every recruiter belongs to one company.  
**Key Fields:** name, slug (unique), industry, size, timezone, currency, careerPageUrl, subdomain, pipelineStages (embedded array), apiKeys (embedded: geminiApiKey, email credentials), billing (embedded: plan, status, cycle).  
**Relationships:** Parent of User, Job, Application, Interview, Offer, HireLetter, Assignment, Message, Notification, AISummary, Template.

#### Job
**Purpose:** Job posting with configurable application form and scoring weights.  
**Key Fields:** title, department, location, employmentType, remoteType, salary range, description, requirements[], skills[], niceToHaveSkills[], status, applicantCount, scoringWeights (skills:40, experience:40, education:20), slug, applicationFormConfig (embedded form field config).  
**Relationships:** Belongs to Company. Referenced by Application, Interview, Offer, HireLetter, Assignment.

## 11. Authentication & Security Analysis

### JWT System (Recruiters)

**Tokens:**
- **Access Token:** 60-minute expiry, signed with `JWT_SECRET`, contains `{ userId, companyId, role }`
- **Refresh Token:** 7-day expiry, signed with `JWT_REFRESH_SECRET`, contains `{ userId }`, stored in `User.refreshTokens[]` array (supports multiple sessions)
- Both stored as **httpOnly cookies** (`secure: true` in production, `sameSite: 'strict'`)
- Access token also returned in response body (for localStorage usage in some client flows)

**Flow:**
1. Login → Generate both tokens → Set cookies → Return in body
2. Request → Read cookie or Authorization header → `verifyAccessToken()` → Extract claims
3. Token expired → `POST /api/auth/refresh` → Validate refresh token against DB array → Issue new access token
4. Logout → Remove refresh token from DB array → Clear cookies

### Candidate Portal Auth (JWT via jose)
- Candidate token stored as `candidate_token` cookie
- Uses `jose` library (Edge-compatible) for verification
- Token payload: `{ candidateId }`

### Google OAuth
- Backend verifies Google ID token via `OAuth2Client.verifyIdToken()` — no client-side trust
- Auto-creates company + user on first Google login

### Route Protection (Next.js Middleware)
`middleware.ts` checks `accessToken` cookie on all protected routes at the Next.js edge layer. Redirects to `/login` if absent. Redirects logged-in users away from auth pages.

### Password Security
- bcryptjs with `genSalt(10)` — 1024 iterations
- Password never stored in plaintext
- Password reset tokens: `crypto.randomBytes(4)` → bcrypt-hashed before storage

### Multi-Tenant Isolation
Every database query filters by `companyId` extracted from the JWT. No route accesses data without this claim. Cross-company data leakage would require a compromised JWT.

### DOMPurify
All user-generated and AI-generated HTML is sanitized before rendering. Prevents XSS.

### Security Strengths ✅
- httpOnly cookies prevent XSS token theft
- Multi-tenant isolation at API layer via JWT claims
- bcrypt password hashing with proper salt rounds
- Google token verification server-side (not client-trusted)
- CORS restricted in socket server comment (wildcard in dev, should restrict in prod)
- Input validation via Zod on all forms

### Security Concerns ⚠️
- CORS is set to `'*'` in `socket-server/index.js` (fine for dev, must restrict in production)
- Access token returned in response body AND set as cookie (dual storage adds risk surface)
- No rate limiting on login endpoint (brute force possible)
- JWT secret has a hardcoded fallback (`'fallback_secret'`) — critical to override in production
- `GEMINI_API_KEY` in `.env` does not start with 'AIza' (the code warns about this)
- Uploaded file handling has no auth middleware in UploadThing core (`userId: "guest"` placeholder)
- `next.config.mjs` allows images from `**` (any hostname) — should restrict to specific CDN domains

## 12. AI Features Analysis

TalentIQ integrates Google Gemini 2.5 Flash in **5 distinct use cases**:

### AI Feature 1: Resume Parsing
**Purpose:** Extract structured data (skills, companies, education) from resume text.  
**Model:** `gemini-2.5-flash`  
**Implementation:** `parseResumeWithGemini()` in `src/core/ai/resume-parser.ts`  
**Prompt Engineering:** Simple extraction prompt with strict JSON schema (`responseMimeType: 'application/json'`, `responseSchema` with required fields)  
**Data Flow:** Frontend parses PDF → extracts text → sends to `POST /api/candidates` → called during application submission → results stored in `Candidate.extractedSkills/extractedCompanies/extractedEducation`  
**Output:** `{ skills: string[], companies: string[], education: string[] }`

### AI Feature 2: Candidate Scoring (Core AI)
**Purpose:** Score candidate 0–100 against job requirements across 4 dimensions.  
**Model:** `gemini-2.5-flash`  
**Implementation:** `scoreCandidateWithGemini()` in `src/core/ai/candidate-scorer.ts`  
**Prompt Engineering:**
- System context: "You are an expert technical recruiter"
- Injects: job title, department, description, required skills, requirements, full resume text
- Response schema enforces: `score, skillsMatch, experienceMatch, educationMatch, keywordsMatch, strengths[], gaps[], reasons[{text, positive}]`
- `responseMimeType: 'application/json'` forces structured output — no free-form text  
**Data Flow:** Resume text + job details → Gemini → structured JSON → stored in `Application.aiScore`  
**Output:** Full score breakdown with actionable strengths/gaps/reasons

### AI Feature 3: Job Description Generation
**Purpose:** Auto-write a professional job description from title, department, and type.  
**Model:** `gemini-2.5-flash`  
**Implementation:** `POST /api/jobs/generate-description`  
**Prompt Engineering:** Clear instructions: 1 overview + 3–5 responsibilities + 3–5 requirements. Critical instruction: "Do NOT use any Markdown formatting." Post-processing strips any escaped markdown with regex.

### AI Feature 4: Hire Letter Generation
**Purpose:** Write a complete, professional formal hire letter.  
**Model:** `gemini-2.5-flash`  
**Implementation:** `POST /api/ai/hire-letter-suggest`  
**Prompt Engineering:** Detailed structure instruction with 7 required sections. Graceful fallback template if Gemini fails.

### AI Feature 5: Weekly Analytics Summary
**Purpose:** Write a 2–3 paragraph executive summary of the company's hiring pipeline.  
**Model:** `gemini-2.5-flash`  
**Implementation:** `GET /api/analytics/summary`  
**Prompt Engineering:** Injects real pipeline metrics. Asks for Markdown format with recommendations. Cached in `AISummary` model to avoid re-calling Gemini every request.

### AI Architecture Notes
- All AI calls use `@google/genai` SDK (not the OpenAI SDK which is also in dependencies)
- Gemini client initialized once in `src/lib/gemini.ts` — API key validated on startup
- Structured output via `responseSchema` avoids JSON parsing failures
- Graceful degradation: all AI functions have fallback returns if Gemini fails
- The `src/lib/ai/prompts.ts` file defines Zod schemas and system prompts for an older/planned AI feature (bias detection in job descriptions — `DetectBiasSchema` — **not yet wired to an active endpoint**)

## 14. Video Calling System Analysis

### Architecture Overview
TalentIQ implements a **serverless-friendly WebRTC system** that avoids the need for a persistent WebSocket server in production:

**Signaling Transport:** HTTP polling against MongoDB (the Signal collection)  
**Media Transport:** Direct browser-to-browser WebRTC peer connection  
**ICE/STUN:** Google STUN servers (no TURN servers — NAT traversal may fail in corporate environments)

### Data Flow

```
Peer A (Interviewer)              MongoDB Signal DB              Peer B (Candidate)
     |                                    |                              |
     |── JOIN ROOM ──────────────────────►|                              |
     |── POST /signal (user-connected) ──►|                              |
     |                                    |◄── GET /signal (poll 2s) ───|
     |                                    |      returns user-connected   |
     |◄──────────────────────────────────|── createOffer ──────────────►|
     |── POST /signal (offer) ───────────►|                              |
     |◄── GET /signal (poll 2s) ─────────|                              |
     |    returns offer                   |                              |
     |── createAnswer ──────────────────►|                              |
     |── POST /signal (answer) ──────────►|                              |
     |◄──────────────────────── [ICE candidates exchanged via polling] ──|
     |═════════════════════ DIRECT P2P WebRTC Connection ═══════════════|
```

### Key Components

**`useWebRTC.ts` (334 lines) — Core Hook:**
- Manages `RTCPeerConnection` lifecycle
- Implements polling loop (every 2 seconds) via `setInterval`
- Processes signals in order: user-connected → offer → answer → ice-candidate → disconnect
- `processedSignals` Set prevents duplicate signal processing
- `pendingIceCandidates` array buffers ICE candidates until remote description is set
- Handles track replacement for mic/camera toggle and screen share

**`meet/[interviewId]/page.tsx` — Video Room UI:**
- Accesses local camera/microphone via `navigator.mediaDevices.getUserMedia`
- Renders local video (mirrored) + remote video placeholder
- Controls: mute, video toggle, screen share, leave call
- Note: This page uses local state only — the full WebRTC peer connection is in `useWebRTC.ts`; the page itself appears to be a simpler demo version without wiring to `useWebRTC`

**`socket-server/index.js` — Alternative Signaling:**
- Express + Socket.io server running on port 3001
- Handles same WebRTC events as the DB-polling approach
- Both systems coexist — Socket.io is for real-time signaling, DB-polling is for serverless-compatible signaling

### WebRTC Capabilities
✅ Local video/audio stream  
✅ Remote video/audio stream  
✅ Microphone toggle (with track replacement)  
✅ Camera toggle (with track replacement)  
✅ Screen sharing (with track replacement back to camera)  
✅ Graceful call end  
✅ Peer disconnect handling  
⚠️ No TURN server — may fail behind strict NAT/firewalls

## 16. Scalability Review

### Current Scalability

**Strengths:**
- Next.js serverless API routes scale horizontally automatically on Vercel
- MongoDB Atlas provides managed scaling
- Inngest handles email queue with concurrency controls
- DB-backed WebRTC signaling works in serverless (no sticky sessions needed)
- TanStack React Query provides client-side caching, reducing API load
- Virtual scrolling handles large candidate lists without DOM explosion

**Bottlenecks:**

1. **WebRTC Polling (2-second interval):** If 100 interviews run simultaneously, that's 200+ polling requests/second just for signaling. MongoDB Signal queries would become expensive. A Redis pub/sub or WebSocket solution would be needed at scale.

2. **AI Scoring on Application Submit:** The candidate application endpoint calls Gemini synchronously during HTTP request. With high application volume, this creates latency (2–5 seconds) and potential timeouts. This should be moved to a background job (Inngest function).

3. **Analytics Summary Generation:** `GET /api/analytics/summary` loads ALL applications, jobs, and candidates for the company into memory for cross-job matching. For companies with 10,000+ candidates, this becomes slow and memory-intensive. Needs MongoDB aggregation pipeline optimization.

4. **No Database Indexes:** Only `Notification` (recipientUserId, companyId), `Signal` (roomId), and `Message` (candidateId) have explicit indexes defined. Heavy queries on `Application.find({ companyId })` and `Job.find({ company })` would benefit from compound indexes.

5. **Socket.io Server:** Single-process Node.js — cannot scale horizontally without Redis adapter.

6. **No Caching Layer:** No Redis/Memcached for caching frequently accessed data (company settings, job lists).

### Recommendations for Scale
1. Move AI scoring to Inngest background function
2. Add MongoDB compound indexes: `Application (companyId + stage)`, `Job (company + status)`, `Interview (companyId + scheduledAt)`
3. Replace polling-based WebRTC signaling with Redis pub/sub or Ably/Pusher for real signaling
4. Add pagination to candidates and applications list APIs (currently returns all)
5. Enable Pusher (already installed) for real notification delivery
6. Add API rate limiting (express-rate-limit or Upstash)

## 22. Technical Interview Preparation

---

**Q1: Walk me through your AI scoring pipeline. How does it actually work end-to-end?**

**Ideal Answer:** When a candidate submits their application, the frontend has already parsed the resume PDF using `pdf-parse` or `mammoth` (for DOCX) into raw text. This text, along with the job's title, description, required skills, and requirements, is sent to my backend API. The backend calls `scoreCandidateWithGemini()` which uses the `@google/genai` SDK with `responseMimeType: 'application/json'` and a strict `responseSchema`. This forces Gemini to return a perfectly structured JSON object — no free-form text that could fail JSON parsing. The schema enforces 7 required fields: score, skillsMatch, experienceMatch, educationMatch, keywordsMatch, strengths array, and gaps array. The returned object is stored directly in the Application document's `aiScore` embedded sub-document. If Gemini fails, a safe fallback with zero scores is returned.

**Technical Explanation:** The key innovation here is using Gemini's structured output feature (`responseSchema`) instead of asking for JSON in the prompt text. This is more reliable because Gemini's inference pipeline enforces the schema at the token level, not just as a best-effort instruction.

---

**Q2: How did you implement WebRTC without a persistent server?**

**Ideal Answer:** Traditional WebRTC requires a persistent WebSocket server for signaling. Instead, I used MongoDB as the signal store. When Peer A joins a room, it POSTs a `user-connected` signal to `/api/interviews/room/[roomId]/signal`. Peer B polls this endpoint every 2 seconds. When it sees the signal, it creates an offer, POSTs it. Peer A polls, sees the offer, creates an answer, POSTs it. ICE candidates are exchanged the same way. The actual video/audio data flows directly peer-to-peer via WebRTC — only the signaling handshake goes through the server. Signal documents have a 24-hour MongoDB TTL index, so they auto-delete. This entire system works on serverless infrastructure like Vercel.

**Technical Explanation:** The tradeoff vs real WebSockets is latency — up to 2 seconds for the connection to establish. For video interviews (non-latency-sensitive for connection setup), this is acceptable. For production at scale, I'd switch to Ably or Redis pub/sub for real-time signaling.

---

**Q3: How does your multi-tenant isolation work?**

**Ideal Answer:** Every Mongoose model has a `companyId` field. Every API route extracts `companyId` from the JWT payload (set at login from the user's `company` field). Every database query includes `companyId: decoded.companyId` as a filter. So even if someone bypasses the frontend, they can only access data from their own company — they can't access another company's candidates or jobs. The JWT is verified server-side on every request.

---

**Q4: Explain your authentication architecture.**

**Ideal Answer:** I have two completely separate auth systems. The first is for recruiters: email+password login uses bcryptjs for hashing (salt rounds 10), generates a 60-minute JWT access token and 7-day JWT refresh token, both stored as httpOnly cookies. The refresh token is also stored in the User document's `refreshTokens` array so I can invalidate specific sessions on logout. Google OAuth is supported — backend verifies the Google ID token using `google-auth-library`'s `OAuth2Client.verifyIdToken()`. The second system is for candidates: when a candidate first applies, a cryptographically random 8-character token is generated with `crypto.randomBytes`, bcrypt-hashed and stored, then returned once to the frontend. The candidate uses this token to log into their portal. Their session uses a separate `candidate_token` cookie verified with `jose`.

---

**Q5: What is your cross-job matching algorithm?**

**Ideal Answer:** It's an O(n×m) algorithm where n is the number of open jobs and m is the number of candidates in the company. For each job, I extract its required skills and requirements into a lowercase keyword array. For each candidate who has NOT already applied to that job, I extract their AI-parsed skills into lowercase strings and compute intersection using `Array.filter` with `String.includes`. The match score is `(matchedKeywords.length / requiredKeywords.length) * 100`, capped at 100. If the score is ≥75, it's recorded as a cross-match. These are surfaced in the weekly AI insights. The algorithmic complexity is manageable for SMBs (hundreds of candidates × tens of jobs), but would need MongoDB aggregation pipelines or dedicated search infrastructure at enterprise scale.

---

**Q6: How does your Kanban drag-and-drop work technically?**

**Ideal Answer:** I use `@dnd-kit/core` with `@dnd-kit/sortable`. Each column is a droppable container. Each candidate card is a draggable item. When a card is dragged to a new column, the `onDragEnd` callback receives the `active` item and `over` container IDs. I optimistically update the Zustand `candidates.store` immediately for instant UI feedback, then fire a `PATCH /api/applications/[id]` to persist the stage change. If the API call fails, I revert the optimistic update. I also use `@tanstack/react-virtual` to virtualize the card list within each column, so hundreds of cards don't create DOM nodes until they're in the viewport.

---

**Q7: How does your email system work?**

**Ideal Answer:** I use Gmail via OAuth2 with Nodemailer. The email credentials (clientId, clientSecret, refreshToken) are stored in environment variables and configured on the Nodemailer transport. Email sending is decoupled from HTTP requests via Inngest. When an event needs to trigger an email, I fire an Inngest event (`email/send.urgent` or `email/send.standard`). Inngest's cloud service receives this, queues it, and executes the corresponding function which calls my `sendEmail()` wrapper. The `send-standard-email` function has `concurrency: { limit: 1 }` to prevent Gmail rate limiting by sending emails one at a time.

---

**Q8: How do you handle resume file uploads?**

**Ideal Answer:** I use UploadThing as the file upload service. The UploadThing router in `/api/uploadthing/core.ts` defines a `resumeUploader` endpoint that accepts PDF files up to 4MB. UploadThing handles chunked upload, CDN storage, and returns a URL. The frontend uses `@uploadthing/react`'s components. After upload, the resume URL is attached to the candidate's application form submission. The frontend also parses the PDF client-side using `pdf-parse` to extract text for AI scoring — this way the text extraction happens before submission and the AI score is available instantly when the form is submitted.

---

**Q9: How does your notification system work?**

**Ideal Answer:** Notifications are created server-side as MongoDB documents whenever key events occur — a new candidate applies (in `POST /api/candidates`), a message is sent (in `POST /api/messages`), etc. The frontend polls `GET /api/notifications` via TanStack Query with a polling interval. The notification model has `recipientUserId` which can be a specific user ID or the string `'all'` for company-wide notifications. Queries filter by `companyId` and `recipientUserId: { $in: [userId, 'all'] }`. Currently this is polling, but I have Pusher installed for upgrading to real-time WebSocket delivery.

---

**Q10: Walk me through your application form and how it's configurable.**

**Ideal Answer:** The Job model has an `applicationFormConfig` embedded object with boolean flags for each possible form field: `requireFullName`, `requireMobileNumber`, `requireDate`, `requireLinkedin`, `requireGithub`, `requirePortfolio`, `requireResume`, `requirePassportPhoto`, `requireSignature`, plus a `customQuestions` string array. When a recruiter creates a job, they can toggle which fields to show. The `ApplicationForm.tsx` component (27KB!) reads this config from the job and conditionally renders the appropriate fields. This allows each job to have a tailored application form without any backend changes.

---

**Q11: How do you handle the candidate portal token system?**

**Ideal Answer:** On first application, the backend generates a random 8-character hex token using `crypto.randomBytes(4).toString('hex').toUpperCase()`. It bcrypt-hashes this token with salt 10 and stores the hash in `Candidate.portalToken`. The plain token is returned once in the API response. The candidate uses this token to log into the portal (`POST /api/portal/auth`). The backend finds the candidate by email, uses `bcrypt.compare(plainToken, candidate.portalToken)` to verify, and issues a separate `candidate_token` JWT cookie. This is similar to how single-use invitation tokens work in many SaaS systems.

---

**Q12: What database design decisions did you make for the timeline feature?**

**Ideal Answer:** The `Application` model has a `timeline` array of embedded event objects (`{ event: string, date: Date, type: enum }`). Each significant action appends to this array — stage changes, offers sent, assignments sent, hire letters sent. This is a document-embedded event log rather than a separate collection, which is appropriate for this scale since timelines typically stay small (10–20 events per application). It trades storage efficiency for query simplicity — you get the full timeline in a single document read.

---

**Q13: How do you prevent double-booking in interview scheduling?**

**Ideal Answer:** The `POST /api/interviews` endpoint first finds all existing non-cancelled interviews for the same company on the same calendar day. For each existing interview, it computes the end time (`scheduledAt + duration * 60000`). Then it checks if the proposed interview's time range `[proposedStart, proposedEnd]` overlaps with any existing range using the overlap condition: `proposedStart < existingEnd && proposedEnd > existingStart`. If any overlap is found, it returns 409 Conflict. It also validates that the proposed start time is between 9:00 AM and 5:00 PM (with a 4:15 PM cutoff to accommodate the 45-minute fixed duration).

**Q14: How did you think about security in this project?**

**Ideal Answer:** Several layers: (1) httpOnly cookies prevent XSS from stealing tokens; (2) JWT `companyId` claim scopes every database query — cross-tenant access is architecturally prevented; (3) bcrypt with salt rounds 10 for password hashing; (4) Google token verified server-side using their official SDK; (5) All user-generated and AI-generated HTML passes through DOMPurify before rendering; (6) Zod validation on all form inputs; (7) MongoDB Signal collection auto-deletes WebRTC signals after 24 hours. Known improvements needed: rate limiting on auth endpoints, restricting CORS wildcard in the socket server, restricting image hostname whitelist in Next.js config.

**Q15: Why did you choose Next.js App Router over a separate frontend + backend?**

**Ideal Answer:** The App Router allowed me to colocate API routes with the frontend in a single codebase. This reduces infrastructure complexity — one deployment, one set of environment variables, one build pipeline. React Server Components let me render initial page content server-side for better SEO on marketing pages. The Edge Middleware intercepts routes at the CDN level without booting Node.js. The tradeoff is that the socket-server has to run as a separate process since WebSocket servers can't live inside Next.js serverless functions.

**Q16: How does the analytics summary get generated?**

**Ideal Answer:** `GET /api/analytics/summary` first checks if an `AISummary` document exists for the current Monday (week start). If yes, it returns the cached version — Gemini is only called once per week per company. If not, it aggregates data from Application, Job, and Candidate collections: counts candidates per pipeline stage, calculates hire rate and avg AI score, runs the cross-job matching algorithm, then constructs a detailed prompt with this data and calls Gemini for a Markdown narrative summary. The result is upserted into `AISummary` (indexed uniquely on `companyId + weekOf`). There's also a `force=true` query param to bypass the cache and regenerate.

**Q17: How does your state management work across the app?**

**Ideal Answer:** I use two layers. Zustand for synchronous client state: auth (current user), UI state (sidebar open/closed), and in-memory domain data (for demo mode: interviews, messages, offers, settings). TanStack Query for server state: it handles fetching, caching, background refetch, and loading/error states for jobs, candidates, applications, notifications. This separation is intentional — Zustand is for things that don't need server sync, TanStack Query is for things that need to stay in sync with the backend. The two rarely interact, which keeps the data flow clear.

---

**Q18: Why did you build your own video calling rather than using an SDK?**

**Ideal Answer:** The main reason was to keep the system self-contained and demonstrate real WebRTC knowledge. Using Agora or Twilio Video would hide the complexity. By implementing the offer/answer/ICE exchange manually, I have full control over the signaling and can use any transport layer — in this case, MongoDB polling. It also eliminates a per-minute per-participant cost that SDK providers charge. The tradeoff is that without TURN servers, the system may fail in corporate firewall environments. For production, I'd add a TURN server (Twilio Network Traversal Service or Coturn).

**Q19: How did you implement the AI bias detection feature?**

**Ideal Answer:** I designed the `DetectBiasSchema` and `DETECT_BIAS` system prompt in `src/lib/ai/prompts.ts` — the intent was a feature where recruiters paste their job description and Gemini analyzes it for biased or exclusionary language, returning a bias score (0–100), list of problematic phrases with suggestions, and overall DEI feedback. However, this feature is **not yet wired to an active API endpoint or UI**. It's defined as a schema and prompt but not fully implemented. I designed it as the next feature to build.

**Q20: What would you change if you had to scale this to 10,000 companies?**

**Ideal Answer:** Several things: (1) Move AI scoring to an Inngest background job — don't block the HTTP response for 3–5 seconds; (2) Add compound MongoDB indexes on `Application(companyId, stage)`, `Job(company, status)`, `Interview(companyId, scheduledAt)`; (3) Add pagination to the candidates and applications list endpoints — currently they return all records; (4) Replace the 2-second polling WebRTC signaling with Ably or Redis pub/sub; (5) Activate Pusher for real-time notifications instead of polling; (6) Add Redis caching for company settings and job lists (read-heavy, rarely change); (7) Add rate limiting on auth and AI endpoints; (8) Replace the Socket.io server's single process with a Redis adapter for horizontal scaling.

## 24. Hidden Features Discovery

### 1. DB-Backed WebRTC Signaling
**Why it's valuable:** Most developers use Socket.io or an external service (Ably, Pusher) for WebRTC signaling. Using MongoDB with TTL indexes as the signal store makes the system fully deployable on serverless infrastructure. This is a clever architectural pattern that eliminates an entire class of infrastructure.

### 2. Structured JSON Output from Gemini
**Why it's valuable:** Using `responseSchema` with `responseMimeType: 'application/json'` instead of asking for JSON in the prompt text makes AI scoring deterministic and failure-resistant. The AI model is constrained to return exactly the right shape of data — not a best effort.

### 3. Candidate Portal Token System
**Why it's valuable:** The 8-character cryptographically random token (bcrypt-hashed server-side) creates a secure, frictionless candidate experience. Candidates don't need to create accounts or remember passwords — they use a short token they can bookmark or screenshot. This is similar to how magic link systems work.

### 4. Auto-Seeding Demo User
**Why it's valuable:** The login endpoint contains this logic:
```typescript
if (!user && email === 'demo@talentiq.com' && password === 'Demo@123') {
  // Auto-create company + user
}
```
This means the system creates a demo account the first time someone tries to log in with those credentials. Elegant for demos and portfolio presentations.

### 5. Dual Token Read (Cookie + Authorization Header)
**Why it's valuable:** Every API route reads the JWT from both the `Authorization: Bearer` header and the `accessToken` cookie. This means the same API works for both traditional server-side requests and SPA (browser fetch) requests without any configuration change.

### 6. Application Timeline System
**Why it's valuable:** The embedded `timeline` array in Application acts as an audit log. Every significant event (stage change, offer, assignment, hire letter) is appended with a typed enum. This gives recruiters a complete history of every touchpoint with a candidate without a separate audit log table.

### 7. `effect` Library (Unused but Present)
**Why it's interesting:** The `effect` package (3.21.3) — a functional programming library for TypeScript — is in dependencies but not used in any reviewed code. This suggests future plans for functional error handling or typed effects in the codebase.

### 8. Bias Detection Schema (Planned Feature)
`src/lib/ai/prompts.ts` contains a fully defined `DetectBiasSchema` and `DETECT_BIAS` system prompt. The design is complete — score 0–100 for DEI inclusivity, list problematic phrases with suggestions. It's wired to no endpoint yet, making it a ready-to-ship feature.

### 9. Company-Level Gemini API Key Override
`Company.apiKeys.geminiApiKey` allows companies on the growth/enterprise plan to use their own Gemini API key instead of the platform's. This is a sophisticated SaaS design pattern for cost management.

### 10. Virtual Scrolling in Kanban
Using `@tanstack/react-virtual` inside a DnD Kit board is non-trivial. Most Kanban implementations render all cards. This implementation handles enterprise-scale candidate volumes without DOM bloat.
